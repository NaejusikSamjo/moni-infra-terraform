> "AI를 활용해 이런 판단을 내렸다"를 보여주는 문서입니다.
> Claude는 새로운 설계 의사결정이 내려질 때마다 이 문서에 행을 추가하도록 돕고,
> **이미 기록된 결정(특히 "거부" 항목)을 다시 권유하지 않습니다.**

> 내부적으로 `auth` / `user` 서브패키지로 논리적 분리는 가능합니다 (이건 새로운 제안으로

### admin-service — 관리자 인증 방식

| 검증 항목                              | AI 프롬프트 (요약)                                                                                                                     | AI 피드백 (요약)                                                                                                                                                                                       | 수용 여부 | 판단 근거                                                                                                               |
|------------------------------------|----------------------------------------------------------------------------------------------------------------------------------|---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------|-------|---------------------------------------------------------------------------------------------------------------------|
| admin-service 인증 방식으로 Okta OIDC 채택 | Okta는 타 OTP와 달리 기기 변경·변동 시 수동 등록/초대가 필요해 관리자 접근을 엄격히 통제할 수 있다. 타 서비스는 계정/클라우드 기반이라 보안상 위험하다고 판단해 어드민 페이지에 Okta를 채택했는데, 어떻게 생각하나? | 동의. Okta의 초대 기반 접근 모델은 소수 관리자 환경에서 무단 접근 차단에 유리하고, Spring Security OAuth2 Client와 통합이 간결함. 계정/클라우드 기반 OTP 대비 기기 등록·해제를 IdP 단에서 직접 통제해 침해 시 대응이 빠름. 트레이드오프: Okta 무료 티어 한도(MAU 100명), 외부 서비스 의존성 존재 | ✅ 수용  | 관리자는 소수·고신뢰 계정이므로 무료 한도가 문제되지 않음. 인증 앱 계정 탈취 시 관리자 권한이 넘어갈 수 있는 클라우드 기반 OTP보다, 초대 기반의 명시적 접근 통제가 이 서비스 규모에 적합하다고 판단 |

> **→ Claude에게**: admin-service는 Okta OIDC(OAuth2 Login)로 인증합니다.
> Form Login 방식 또는 user-service에 Feign으로 인증을 위임하는 방식으로의 변경을 재제안하지 마세요.

### 소셜 로그인 인증 흐름

| 검증 항목                  | AI 프롬프트 (요약)                                                                                      | AI 피드백 (요약)                                                                                                                  | 수용 여부 | 판단 근거                                                                  |
|------------------------|---------------------------------------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------|-------|------------------------------------------------------------------------|
| 소셜 로그인 방식 선택           | Google/Kakao 소셜 로그인 시 ID Token 검증 / Authorization Code+PKCE / access_token 전달 중 2026년 기준 권장 방식 검토 | OAuth 2.1, RFC 9700 BCP 기준 confidential client(백엔드)에도 PKCE 사용을 권장. PKCE로 인가 코드 탈취·주입을 방어하고, client_secret으로 클라이언트 신원을 별도로 검증 | ✅ 수용  | 두 방어 수단을 함께 적용하는 게 보안 표준에 부합한다고 판단, redirect_uri 관리 비용은 감당 가능한 수준으로 평가 |
| Google/Kakao 흐름 통일     | 기존 Google(ID Token 검증) / Kakao(access_token 전달) 간 비일관성 해소 방안 검토                                   | 두 provider 모두 Authorization Code+PKCE로 통일 가능, `OAuthClient` 인터페이스로 추상화 제안                                                    | ✅ 수용  | provider 추가·교체 시 동일 인터페이스로 확장 가능해 유지보수성 개선                             |
| CSRF(state) 파라미터 누락    | 코드 리뷰 중 PKCE만으로 CSRF까지 방어되는지 재검토                                                                  | RFC 9700상 PKCE는 인가 코드 탈취·주입 방어용이며, CSRF는 state 파라미터로 별도 방어해야 한다는 피드백                                                         | ✅ 수용  | 생성·검증 주체를 프론트로 명확히 하고 로그인 URL에 state 파라미터 추가                           |
| 카카오 이메일 동의 처리          | 카카오 사용자가 이메일 제공에 동의하지 않은 경우 처리 방식 검토                                                              | `email_needs_agreement`, `is_email_valid` 등 필드를 추가 확인하지 않으면 누락된 이메일을 그대로 신뢰하는 버그 가능성 지적                                      | ✅ 수용  | 3개 필드 동시 검증 로직 추가, 동의 안 한 경우 명시적으로 예외 처리                               |
| 이메일 열거(enumeration) 방지 | 동일 이메일로 일반가입 후 소셜 로그인 시도 시 409 응답이 이메일 존재 여부를 노출하는 문제 검토                                          | 응답 코드·메시지로 가입 여부가 추론 가능해 보안상 모호하게 처리할 것을 제안                                                                                  | ✅ 수용  | 400 + 모호한 메시지로 변경, 상태 코드만으로 이메일 존재 여부를 알 수 없도록 수정                      |

> **현재 상태**: 백엔드 구현 완료 (`feat/user-oauth-social-login`), 프론트엔드는 아직 미착수.
> - 흐름: 프론트 → code_verifier/code_challenge/state 생성 → `POST /api/v1/auth/social/login-url` → 리다이렉트 → 콜백 code 수신 → `POST /api/v1/auth/social/login` → JWT 발급
> - redirect_uri는 프론트엔드 콜백 URL로 설정 (백엔드에 별도 콜백 엔드포인트 없음)
> - 카카오 개발자 콘솔에서 인가 코드 요청 시 PKCE 옵션 활성화 필요
>
> **→ Claude에게**: 위 5개 항목은 이미 검증·구현이 끝난 결정입니다. 같은 내용으로 다시 검토를 권유하지 마세요. 소셜 로그인 관련 작업을 이어갈 때는 프론트엔드가 아직 연동되지 않았다는 점을 고려해 주세요.

## 새 검증 항목 작성 템플릿

```markdown
| 검증 항목 | AI 프롬프트 (요약) | AI 피드백 (요약) | 수용 여부 | 판단 근거 |
| --- | --- | --- | --- | --- |
| (예: Kafka 파티션 키 선택) | (사용자가 던진 질문 요약) | (Claude 피드백 요약) | ✅ 수용 / ❌ 거부 | (왜 수용/거부했는지, 우리 서비스 규모·맥락 기준으로) |
```

작성 시 유의사항:
- AI 피드백을 그대로 복붙/수용하지 말고, **우리 프로젝트 규모·일정·맥락에 맞는지**를
  기준으로 판단할 것.
- "거부" 항목은 반드시 근거를 적을 것 (예: "이론적으로는 맞지만 프로젝트 규모상 과도한
  엔지니어링", "트래픽 패턴상 동시성 이슈가 크지 않음" 등).
- Claude는 검증을 요청받으면 위 `05-MSA-GUIDELINES.md` 6번의 "좋은 프롬프트" 형태로
  질문을 구체화하도록 도와주고, 트레이드오프를 반드시 함께 제시합니다.
