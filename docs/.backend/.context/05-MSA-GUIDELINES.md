# 05. MSA 가이드라인 (서비스 분리 / 통신 / 안티패턴 체크리스트)

> 원본: 과정에서 제공된 "MSA 서비스 분리 기준 가이드라인" 문서를 정리. 이 프로젝트는 이미
> 7개 도메인 서비스(+ 4개 공통 인프라 컴포넌트)로 분리된 상태로 시작하지만, 아래 기준은
> 새 기능을 어디에 둘지, 추가 서비스를 분리할지 판단할 때 계속 참고해야 합니다.

---

## 0. 가장 중요한 원칙

> 🚨 "하나의 모놀리식을 만들 수 있는 사람이, 이유가 있어서 나눈 것"이 좋은 MSA입니다.
> 서비스를 5개 이상으로 쪼개면 통신/디버깅/환경구성에 시간을 다 쓰게 됩니다.

이 프로젝트는 이미 10개 모듈(서비스+공통 인프라)로 시작했으므로, **신규 서비스를 추가로
쪼개는 제안은 신중하게** 해야 합니다. 새 기능은 우선 기존 7개 도메인 서비스 중 어디에
속하는지부터 검토하세요.

---

## 1. 서비스 경계를 나누는 기준 (우선순위 순)

아래 질문에 "예"가 많을수록 분리 후보:

1. 독립적으로 배포·확장하고 싶은가? (예: 특정 트래픽만 폭증 → 해당 서비스만 스케일아웃)
2. 데이터 소유권이 명확히 다른가? (서로 다른 테이블 묶음을 주로 다루는가)
3. 변경되는 이유가 다른가? (정책 변경이 서로 무관하게 일어나는가)
4. 장애가 전파되면 안 되는가? (한 서비스가 죽어도 다른 서비스는 동작해야 하는가)

➡️ 한두 개만 "예"라면 나누지 않는다. **나누는 것은 비용**이다.

---

## 2. 서비스 간 통신: 동기 vs 비동기

| 상황                                             | 권장 방식              | 이유                                 |
|------------------------------------------------|--------------------|------------------------------------|
| 즉시 응답에 필요한 정보 확인 (예: 매수 시 보유 현금/종목 가격 확인)      | **동기 (OpenFeign)** | 응답이 바로 필요하고, 실패하면 해당 요청도 실패해야 함    |
| 완료 후 알림·집계·로그 적재 (예: 거래 체결 후 포트폴리오 재계산, 알림 발송) | **비동기 (Kafka)**    | 즉시 응답에 불필요, 후처리가 느려도 사용자 경험에 영향 없음 |

> ✅ 권장 규칙: **"사용자 응답에 꼭 필요한 것만 동기, 나머지는 전부 비동기."**

### 동기 호출(Feign) 시 반드시 고려할 것

- **Timeout 설정**: Feign 기본 타임아웃을 짧게 명시
- **Circuit Breaker / Fallback**: Resilience4j로 상대 서비스 장애 시 대체 응답
- 이 부분은 JMeter 부하 테스트 결과와 직접 연결됨 (`04-INFRA-ARCHITECTURE.md` 7번)

이 프로젝트에서 알려진 동기 호출 예: portfolio-service → ai-service
(`POST /api/v1/ai/portfolio/analysis`, Authorization 헤더 전달, `02-API-SPEC.md` 5번 참고).

## 3. 데이터 정합성: 분산 트랜잭션은 최소화

> ⚠️ 2PC나 완전한 Saga 구현을 강제하지 않음. **개념을 이해하고, 어디가 위험한지 설명할 수
> 있으면 충분**.

- **원칙**: 하나의 트랜잭션은 하나의 서비스 안에서 끝나도록 설계 (서비스 경계 = 트랜잭션 경계)
- 서비스를 걸쳐야 한다면: `트랜잭션 발생 → Kafka 이벤트 발행 → 후처리는 이벤트로(최종적 일관성)`
- 보상 트랜잭션(실패 시 되돌리기)은 도전 영역 — 시도했다면 AI 검증 로그에 근거와 함께 기록

이 프로젝트의 대표 후보: 주식 매수/매도(trade-service) → 계좌 잔액/보유종목 변경 →
포트폴리오 재계산(portfolio-service) → AI 분석 카운트 갱신 등. 이런 흐름을 구현할 때는
"트랜잭션이 어디서 끝나고, 어디부터 이벤트 기반 후처리인지"를 먼저 설계하고 사용자와
공유하세요.

## 4. 흔한 안티패턴 체크리스트 (코드 작성/리뷰 시 자가 점검)

- [ ] 서비스가 5개 이상이다 → 합칠 수 있는지 재검토 (이 프로젝트는 이미 7개 도메인
  서비스이므로, **새로운 서비스 분리는 강한 근거가 없으면 제안하지 말 것**)
- [ ] 두 서비스가 **같은 DB·같은 테이블**을 직접 공유한다 → 분리한 의미 없음(분산 모놀리식).
  특히 `account`/`holding` 중복 문제(`03-ERD.md` 하단 참고) 주의
- [ ] 한 기능을 위해 서비스 3개 이상을 동기 호출로 연쇄 호출한다 → 장애 전파·지연 위험
- [ ] 서비스 간 통신이 전부 동기다 → 비동기로 뺄 수 있는 후처리가 있는지 확인
- [ ] 서비스를 나눈 이유를 한 문장으로 설명할 수 없다 → 나누지 말 것

## 5. AI 설계 검증과 연결하기 (Claude 사용 시 적용)

설계를 마친 뒤, 아래와 같은 형태의 프롬프트로 검증받고 결과를 `06-AI-VALIDATION-LOG.md`에
기록합니다. Claude가 이런 검증 요청을 받으면:

- 무조건 동의하지 말고, 위험 지점과 대안을 트레이드오프와 함께 제시
- 사용자가 수용/거부를 결정하면, 로그에 추가할 행(검증 항목 / 프롬프트 요약 / AI 피드백 요약 /
  수용 여부 / 판단 근거) 작성을 도움

검증 프롬프트 템플릿 예시:

```
나는 [기능/모듈]을 [서비스 A / 서비스 B] 구조로 나눴어.
각 서비스의 책임과 소유 데이터는 다음과 같아: (붙여넣기)
서비스 간 통신은 [예: A→B는 Feign 동기, B→A 알림은 Kafka 비동기]로 설계했어.
이 분리 경계가 적절한지, 장애 전파나 데이터 정합성 측면에서 위험한 지점이 있는지 검토해줘.
무조건 동의하지 말고, 더 나은 대안이 있으면 트레이드오프와 함께 제시해줘.
```

## 6. 좋은 프롬프트 vs 나쁜 프롬프트 (참고용)

설계 검증을 잘 받으려면 설계 의도를 구체적으로 전달해야 합니다.

| 상황       | ❌ 나쁜 프롬프트              | ✅ 좋은 프롬프트                                                                                                               |
|----------|------------------------|-------------------------------------------------------------------------------------------------------------------------|
| 도메인 설계   | "이커머스 시스템 설계해줘"        | "주문-결제 서비스에서 분산 트랜잭션이 필요해. 2PC의 성능 저하 없이 Transactional Outbox 패턴과 Kafka로 데이터 일관성을 보장하는 방식을 검토해줘."                       |
| 성능 최적화   | "N+1 문제 해결해줘"          | "복잡한 연관 관계에서 N+1이 발생해. Querydsl Fetch Join으로 최적화하고, 데이터가 없는 경우를 대비한 Optional 처리와 DTO Projection으로 DB 부하를 줄이는 방안을 검토해줘." |
| 동시성 제어   | "선착순 쿠폰 발급 시스템 만들어줘"   | "선착순 쿠폰 발급이라 동시 요청이 폭주해. DB Lock보다 Redis Sorted Set이나 Lua Script로 원자성을 보장하면서 처리량을 극대화하는 로직을 설계해줘."                      |
| 장애 전파 방지 | "Circuit Breaker 구현해줘" | "외부 PG사 API 호출 타임아웃 시 우리 서비스 전체가 마비될 수 있어. Resilience4j로 Circuit Breaker를 걸고 폴백(Fallback) 메서드 구현까지 포함해줘."               |

## 7. 계층형 아키텍처(4계층) AI 검증 시 확인 항목

- 비즈니스 로직이 응용(Application) 계층에 새어나가지 않고 도메인(Domain) 계층에 응집되어 있는가
- 인프라 계층의 기술 구현이 도메인 계층에 역방향으로 의존하고 있지 않은가 (의존 역전 원칙)
- 표현(Presentation) 계층의 DTO와 도메인 엔티티가 직접 결합되어 있지 않은가
- 응용 계층 서비스가 너무 많은 책임을 갖고 있지 않은가 (Fat Service 안티패턴)
- 외부 시스템 연동(Kafka, Redis 등)이 인프라 계층에 격리되어 있는가

(구체적인 패키지 구조 규칙은 root `CLAUDE.md` 5번 항목 참고)
